# DocReview 触发条件定义

## 适用场景

| 编号 | 场景 | 触发条件 | 输入类型 |
|------|------|----------|----------|
| S1 | PRD预审查 | 产品经理完成PRD后、团队审查前 | PRD文档 |
| S2 | 技术方案验证 | 技术主管完成技术方案后 | 技术方案/实施计划 |
| S3 | 智能体文档审计 | 其他智能体（如SOLO Coder）生成文档后 | 自动化生成的文档 |
| S4 | 质量门禁 | 新功能开发或重构项目启动前 | 任意结构化技术文档 |
| S5 | 修订后重新审查 | 文档修订后验证修改质量 | 修订版文档 |

## 触发条件详细说明

### S1: PRD预审查
**何时使用**：
- 产品经理完成 PRD 初稿
- PRD 提交团队评审前
- 大型需求立项前

**输入**：
- PRD 文档（Markdown 格式）
- 可选：产品背景文档

**预期输出**：
- 结构化审查报告
- 问题列表及修改建议
- 是否可进入开发阶段的建议

### S2: 技术方案验证
**何时使用**：
- 技术方案设计完成后
- 评审会议前
- 关键技术决策确定前

**输入**：
- 技术方案文档
- 可选：PRD、架构设计

**预期输出**：
- 技术可行性评估
- 风险识别
- 建议改进点

### S3: 智能体文档审计
**何时使用**：
- AI 生成的文档需要人工审核前
- 自动化文档质量保证流程中

**输入**：
- AI 生成的文档

**预期输出**：
- 幻觉检测
- 事实性验证
- 完整性评估

### S4: 质量门禁
**何时使用**：
- CI/CD 流程中的文档检查点
- 项目启动前
- 重要里程碑检查

**输入**：
- 任意结构化技术文档

**预期输出**：
- 质量评分
- 问题清单
- 通过/不通过判定

### S5: 修订后重新审查
**何时使用**：
- 文档根据审查意见修订后
- 验证修订质量

**输入**：
- 修订版文档
- 原审查报告

**预期输出**：
- 修订对比摘要
- 遗漏问题检查
- 重新判定结论

## CLI 使用示例

```bash
# 审查 PRD
docreview review --doc-path ./docs/prd.md --max-iterations 5

# 根据任务生成规格并审查
docreview review --task "设计一个用户认证系统"

# 审查技术方案
docreview review --doc-path ./docs/tech-solution.md --verbose
```
